# AI Data Copilot 演示案例设计

> 本文档为 AI Data Copilot 系统提供五种角色、多种场景的演示案例，用于产品演示、功能验证和用户引导。

---

## 角色概览

| 角色 | 定位 | 典型场景 | 使用模式 |
|------|------|----------|----------|
| **数据开发工程师** | 负责数据建模、SQL 开发、问题排查 | 建表咨询、SQL 加工、语法校验、报错排查 | ReAct 模式（大模型生成 SQL 和解答） |
| **业务分析人员** | 负责业务数据分析、指标查询 | 销售查询、用户分析、指标对比 | ReAct / Planner 模式（大模型生成 SQL → 执行 → 返回结果） |
| **数据运维工程师** | 负责 Pipeline 运维、故障排查 | Flink Job 状态、Kafka 消费延迟、日志分析、告警趋势 | TROUBLESHOOT 模式（调用运维工具 → 诊断分析 → 建议操作） |
| **数据管理员** | 负责元数据管理、数据治理 | 表结构查询、血缘追踪、权限管理、Schema 变更历史 | METADATA_QUERY 模式（调用元数据工具 → 结构化返回） |
| **数据产品经理** | 负责数据看板、指标体系、业务洞察 | 指标趋势、下钻分析、用户画像、看板规划 | METRIC_ANALYSIS / CHAT 模式（多步分析 → 业务报告） |

## 案例库

### 角色一：数据开发工程师 —— "张工"

**人物画像**：数据平台开发工程师，负责数据仓库建模、ETL 开发、SQL 性能优化。希望通过 AI 助手快速获得建表建议、SQL 编写帮助和报错排查指导。

> **使用特点**：问题偏技术向，回答以大模型生成的 SQL 代码和文字说明为主，不一定需要实际执行。

#### 案例 1.1：新建订单汇总表

**场景**：张工需要新建一张订单汇总表，不确定字段类型和分区策略。

**用户提问**：
```
我想在 ClickHouse 里建一张订单日汇总表，包含日期、区域、品类、订单数、GMV、下单用户数，按天分区，帮我生成建表语句
```

**预期流程**：
1. 意图识别 → SQL_QUERY
2. 大模型生成 ClickHouse 建表 SQL
3. 返回建表语句及说明

**预期回复示例**：
```sql
CREATE TABLE IF NOT EXISTS copilot.order_daily_summary
(
    order_date   Date        COMMENT '订单日期',
    region       String      COMMENT '区域',
    category     String      COMMENT '品类',
    order_count  UInt64      COMMENT '订单数',
    gmv          Decimal(18,2) COMMENT 'GMV',
    buyer_count  UInt64      COMMENT '下单用户数'
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(order_date)
ORDER BY (order_date, region, category);
```

**验证点**：
- SQL 语法正确（ClickHouse 方言）
- 包含合理的字段类型和注释
- 分区和排序策略合理

---

#### 案例 1.2：复杂 SQL 加工 — 多表关联查询

**场景**：张工需要编写一条关联订单、用户、商品三表的 SQL，用于分析高价值用户的购买偏好。

**用户提问**：
```
帮我写一条SQL，统计每个会员等级中，购买金额前10的商品品类，输出等级、品类、总金额、订单数，按等级和总金额排序
```

**预期流程**：
1. 意图识别 → SQL_QUERY
2. 大模型生成多表 JOIN SQL
3. 可选：实际执行验证 SQL 是否正确

**预期回复示例**：
```sql
SELECT
    u.level        AS member_level,
    o.category     AS product_category,
    SUM(o.amount)  AS total_amount,
    COUNT(*)       AS order_count
FROM orders o
JOIN users u ON o.user_id = u.user_id
WHERE o.status IN ('paid', 'shipped', 'completed')
GROUP BY u.level, o.category
ORDER BY u.level, total_amount DESC
LIMIT 40;
```

**验证点**：
- 多表 JOIN 逻辑正确
- GROUP BY 和 ORDER BY 合理
- 包含状态过滤避免重复计算

#### 案例 1.3：SQL 报错排查

**场景**：张工写了一条 SQL 报错，希望 AI 帮忙分析问题。

**用户提问**：
```
这条SQL报错帮我看看什么问题：
SELECT region, sum(amount) as gmv FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31' GROUP BY region HAVING gmv > 100000 ORDER BY gmv
```

**预期流程**：
1. 意图识别 → TROUBLESHOOT
2. 大模型分析 SQL 语法和可能的错误原因
3. 给出修复建议

**预期回复示例**：
> 这条 SQL 语法本身没有明显错误。可能的报错原因：
> 1. 如果 ClickHouse 报 `Unknown identifier` 错误，检查 `order_date` 字段是否为 Date 类型，字符串比较需要确保格式匹配
> 2. 如果报 `Memory limit exceeded`，建议增加 `LIMIT` 子句或缩小日期范围
> 3. 如果表不在默认数据库中，需要加表名前缀 `copilot.orders`
>
> 建议改写为：
> ```sql
> SELECT region, sum(amount) AS gmv
> FROM copilot.orders
> WHERE order_date >= '2024-01-01' AND order_date <= '2024-12-31'
> GROUP BY region
> HAVING gmv > 100000
> ORDER BY gmv DESC;
> ```

**验证点**：
- 能识别潜在问题
- 给出可操作的修复建议

---

#### 案例 1.4：SQL 性能优化咨询

**场景**：张工有一条慢 SQL，希望得到优化建议。

**用户提问**：
```
这条SQL执行很慢，怎么优化：
SELECT u.user_id, u.city, o.order_id, o.amount, p.product_name
FROM users u
JOIN orders o ON u.user_id = o.user_id
JOIN products p ON o.product_id = p.product_id
WHERE o.order_date >= '2024-01-01'
ORDER BY o.amount DESC
```

**预期流程**：
1. 意图识别 → TROUBLESHOOT
2. 大模型分析性能瓶颈并给出优化建议

**预期回复要点**：
- 建议在 `orders(order_date, user_id, product_id)` 上建索引/优化 ORDER BY
- 建议过滤条件前置，减少 JOIN 数据量
- 建议只查需要的字段，避免 SELECT *
- 考虑按日期分区裁剪

**人物画像**：大数据平台运维工程师，负责 Kafka、Flink 等数据 Pipeline 的日常运维、故障排查和性能调优。希望通过 AI 助手快速定位 Pipeline 异常、分析日志并给出修复建议。

#### 案例 3.1：Flink Job 状态排查

**场景**：王工收到告警，某个 Flink Job 运行异常，希望 AI 帮忙排查。

**用户提问**：
```
order_sync_job 这个 Flink 任务现在状态怎么样，checkpoint 延迟大吗
```

**预期流程**：
1. 意图识别 → TROUBLESHOOT
2. Planner 规划：
   - Step 1: 调用 `flink` 工具查询 Job 状态
   - Step 2: 调用 `flink` 工具查询 Checkpoint 延迟和 Backpressure
3. 大模型整合结果生成诊断报告

**预期回复示例**：
> **order_sync_job 当前状态**：
>
> | 指标 | 值 |
> |------|-----|
> | Job 状态 | RUNNING（存在异常） |
> | 运行时长 | 3 小时 42 分 |
> | Checkpoint 延迟 | 120s（正常值 < 30s） |
> | Backpressure | HIGH |
> | 最近一次 Checkpoint | 失败（超时） |
>
> **诊断**：该 Job 存在严重反压，Checkpoint 延迟高达 120s，最近一次 Checkpoint 因超时而失败。可能原因：
> 1. 下游 ClickHouse 写入瓶颈，导致算子积压
> 2. 数据倾斜，某个 Subtask 处理量远高于其他
>
> **建议操作**：
> - 检查下游 ClickHouse 的写入吞吐和 CPU 使用率
> - 查看各 Subtask 的数据分布，确认是否存在倾斜
> - 临时方案：增大 Checkpoint 超时时间至 5min

**验证点**：
- 正确调用 `flink` 工具获取 Job 状态和性能指标
- 能综合多项指标给出根因分析
- 建议操作具体可执行

#### 案例 3.2：Kafka 消费延迟分析

**场景**：王工发现某个 Topic 消费积压严重，想了解具体原因。

**用户提问**：
```
orders topic 的消费 lag 有多大，是哪个消费组的问题
```

**预期流程**：
1. 意图识别 → TROUBLESHOOT
2. 调用 `kafka` 工具查询 Topic Lag、消费组状态、分区倾斜
3. 大模型分析数据给出结论

**预期回复示例**：
> **orders topic 消费情况**：
>
> | 消费组 | 总 Lag | 状态 | 最大分区 Lag |
> |--------|--------|------|-------------|

#### 案例 5.4：用户画像分析

**场景**：陈经理想了解高价值用户的特征，用于制定运营策略。

**用户提问**：
```
金卡和黑金会员的购买行为有什么特点，和普通用户差多少
```

**预期流程**：
1. 意图识别 → METRIC_ANALYSIS
2. Planner 规划：
   - Step 1: 查询各等级用户的人均订单数、人均消费金额
   - Step 2: 查询各等级用户的品类偏好
   - Step 3: 查询各等级用户的复购率
3. 大模型整合分析生成用户画像

**预期回复示例**：
> **不同等级用户购买行为对比**：
>
> ### 消费能力
>
> | 指标 | 普通用户 | 银卡 | 金卡 | 黑金 |
> |------|---------|------|------|------|
> | 人均订单数 | 2.3 | 4.1 | 8.7 | 15.2 |
> | 客单价（元） | 156 | 285 | 520 | 1,280 |
> | 月均消费（元） | 358 | 1,169 | 4,524 | 19,456 |
>
> ### 品类偏好（TOP 3）
>
> | 等级 | TOP 1 | TOP 2 | TOP 3 |
> |------|-------|-------|-------|
> | 普通 | 食品（42%）| 服装（28%）| 家居（15%）|
> | 银卡 | 服装（35%）| 食品（25%）| 美妆（18%）|
> | 金卡 | 电子产品（38%）| 服装（22%）| 美妆（18%）|
> | 黑金 | 电子产品（52%）| 美妆（20%）| 服装（15%）|
>
> ### 关键发现
> 1. **黑金用户客单价是普通用户的 8.2 倍**，月均消费差距达 54 倍
> 2. 等级越高，**电子产品**占比越高（普通 8% → 黑金 52%）
> 3. 黑金用户复购率 78%，远高于普通用户的 25%
>
> **运营建议**：针对银卡和首次购买电子产品的用户推送高价值商品，促进等级提升。

**验证点**：
- 多维度对比分析（消费能力、品类偏好）
- 数据支撑的洞察结论
- 有业务建议
