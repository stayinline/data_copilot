## 一、项目背景及价值（2:30）

### 1.1 当前问题

**标题：花很多资源建的数据平台，90%的人用不了**

**幻灯片视觉建议：** 一张时间轴，从早上9点到晚上8点，标注不同角色因为"数据触达不到"而发生的问题。背景是一堵灰色的"数据高墙"。

**讲述稿：**

> 各位好。我想先讲一个真实的故事。
>
> 这是阿里九九公司上周一个普通的工作日：
>
> **早上9:00，运营小李**打开BI系统，看到12个菜单，不知道该点哪个。她想知道昨天活动的效果——但她不会写SQL，也看不懂仪表盘。最后她给数据工程师发了一条消息："帮我查一下昨天活动数据。" **等待开始。**
>
> **早上9:30，销售总监老张**要看区域业绩。他找到了分析师，分析师说："前面还有3个需求在排队，预计下午给你。" **等待继续。**
>
> **早上10:00，产品经理小陈**准备下午的版本评审会，他想知道新功能上线后的用户使用率。但数据分散在埋点系统、A/B测试平台和用户反馈工具里，他翻了一个小时也没拼出完整结论。**数据到处都是，但连不起来。**
>
> **早上10:30，供应链采购老赵**发现一批关键物料库存告急。他需要查ERP库存、看物流在途、算安全库存天数——但这些数据在三个不同系统里，每次都要手动导出Excel再合并。**等他把数据凑齐，产线已经快断料了。**
>
> **早上11:00，领导开会问**："华东地区上个月业绩达标了吗？" 全场没人能立刻回答。最后说"我下午拉个报表"。**等了一整天。**
>
> **下午2:00，运维小王**发现Kafka消费延迟飙升。他登录Flink控制台，看Prometheus，查日志系统，看告警平台——**来回切换4个系统**，花了30分钟才定位到问题。
>
> **而数据工程师呢？** 他一天收到了80条"帮我查一下"的消息，80%的时间在写重复的SQL。**高级工程师变成了取数工具人。**
>
> 
>
> **这就是现实——花很多资源建的数据平台，90%的人用不了。**
>
> 全公司600个人，只有不到10个人能真正碰到数据。其他590人——运营、销售、产品、采购、管理者——的数据需求，只能通过"喊"给那10个人来解决。

### 1.2 哪个流程低效？哪部分重复劳动最多？

**标题：不是同一种痛，是七种不同的绝望**

**幻灯片视觉建议：** 七个人物卡片，展示角色 + 核心痛点 + 当前解决方式。

| 角色        | 他的问题 | 当前解决方式 | 痛点本质 |
|-----------|---|---|---|
| **运营**    | "昨天活动效果怎么样？" | 发消息给数据工程师，等半天 | **不会查** |
| **销售总监**  | "各区域业绩排名？" | 找分析师排期，等3天 | **够不到** |
| **产品经理**  | "新功能用户使用率？" | 翻埋点系统、A/B平台、反馈工具，拼不出完整结论 | **连不起来** |
| **供应链采购** | "关键物料库存告急" | 手动导出三个系统Excel合并，凑齐时已断料 | **来不及** |
| **管理者**   | "GMV为什么下降了？" | 开会讨论，拍脑袋猜 | **看不懂** |
| **运维工程师** | "Kafka延迟谁的问题？" | 切换4个系统，手动排查30分钟 | **查不全** |
| **数据工程师** | "帮我写这个SQL" | 每天写80遍重复SQL | **被绑架** |

**讲述稿：**

> 刚才的故事里，每个角色的痛点其实不一样。我拆开来说：
>
> **运营的痛点是"不会查"**——BI系统太复杂，SQL不会写，数据就在眼前但拿不到。
>
> **销售总监的痛点是"够不到"**——需要排期、需要等，数据获取的链路太长。
>
> **产品经理的痛点是"连不起来"**——数据散在埋点、A/B测试、用户反馈等多个系统，手动拼凑永远缺一块。
>
> **供应链采购的痛点是"来不及"**——跨系统手动导出合并，等数据凑齐了，产线已经断料。数据有延迟，决策就迟到。
>
> **管理者的痛点是"看不懂"**——就算拿到数据报表，也无法回答"为什么"。数据是静态的，问题是动态的。
>
> **运维的痛点是"查不全"**——数据分散在Flink、Kafka、日志系统、告警平台，手动串联效率极低。
>
> **数据工程师的痛点是"被绑架"**——80%时间在做重复劳动，没精力做真正有价值的事。
>
> 七种角色，七种痛点。但本质是一个问题：**现有的数据平台，没有一种能同时服务所有人。**
>
> 这不是工具的问题，**这是范式的问题。** 只要"人必须学工具"这个逻辑不变，再强的数据平台，也只是一堵更高的墙。

## 二、项目创意：AI 创新玩法（1:00）

### 标题：不是"更好的工具"，是"不同的物种"

**幻灯片视觉建议：** 左边表格"Text-to-SQL 覆盖范围 vs AI Copilot 覆盖范围"——Text-to-SQL 只有"查数据"一行打勾，AI Copilot 6行全勾。右侧列出三大创新玩法。中间一句话："Text-to-SQL 只是翻译器，我们是自主智能体。"

| 问题类型 | Text-to-SQL | AI Data Copilot |
|---|---|---|
| 查数据 | ✅ | ✅ |
| 趋势分析 | ❌ | ✅ 自动分析 |
| 归因下钻 | ❌ | ✅ 自动4层下钻 |
| 故障排查 | ❌ | ✅ 全链路诊断 |
| SQL编写/诊断 | 部分 | ✅ 自动生成+诊断 |
| 元数据查询 | ❌ | ✅ 秒级返回 |

**讲述稿：**

> 有人可能会问：这不就是 Text-to-SQL 吗？
>
> **如果只是 Text-to-SQL，那不值一提。** Text-to-SQL 本质是一个翻译器——把你的话翻译成SQL。但现实问题远不止"查一下"这么简单。
>
> 看这张图——**Text-to-SQL 只能解决最上面一行**，归因分析、故障排查、SQL诊断、元数据查询，它全干不了。
>
> AI Data Copilot 是一个**自主智能体**，创新在三个层面：
>
> **1. 多 Agent 协作**——三层意图识别（关键词 → Embedding → LLM兜底），覆盖5种意图，自动走不同处理路径。
>
> **2. Workflow 自动化**——基于 LangGraph 的 DAG 任务规划。"GMV为什么下降"——它会自己决定先对比整体、按区域拆、按品类拆、交叉验证，**全程自动，多步执行**。
>
> **3. AI 自动决策**——主动分析、主动下钻，同时内置五层风暴防护——调用次数、深度、去重、超时、熔断，**确保AI不会失控**。
>
> **一句话总结：Text-to-SQL = 翻译一条SQL，AI Copilot = 搞定所有事。**

## 三、解决方案及技术实现（2:00）

### 标题：看似魔法，背后是精心设计的三层引擎

**幻灯片视觉建议：** 三层引擎架构图 + 数据流。
```
用户输入 → 意图识别 → 任务规划(ReAct循环) → 工具执行 → 结果总结
```
底层架构图：
```
┌─────────────────────────────────┐
│  🧠 认知层：意图理解引擎            │
│     规则匹配 → 语义向量 → LLM推理   │
├─────────────────────────────────┤
│  🎯 决策层：自主规划引擎            │
│     LangGraph状态机 + 五层风暴防护  │
├─────────────────────────────────┤
│  ⚡ 执行层：工具链引擎               │
│     6大工具 + 4层SQL安全 + 自动修复 │
└─────────────────────────────────┘
```

**讲述稿：**

> 这套系统为什么能做到？因为它的架构不是"调个API"那么简单——而是一个完整的**三层智能引擎**：
>
> **第一层：认知层 — 听得懂**
>
> 三层意图识别：关键词规则毫秒级响应 + 语义向量深度匹配 + 大模型兜底推理。覆盖5种意图类型。
>
> 更关键的是，**它能识别"低质量提问"并主动引导**——如果你问的问题不够明确，它会展示正确的提问方式，教你怎么用。
>
> **第二层：决策层 — 想得好**
>
> 基于 LangGraph 构建的自主规划引擎，采用 DAG 任务分解 + ReAct 循环。它不是按固定流程走，而是**实时决策**：拿到一个结果后，自己判断是继续下钻、还是给出结论、还是调用另一个工具。
>
> 同时内置**五层风暴防护**，确保AI不会失控。
>
> **第三层：执行层 — 干得准**
>
> 6大工具覆盖全场景：`run_sql`（数据查询）、`query_metadata`（元数据与血缘）、`kafka`（Kafka运维）、`flink`（Flink运维）、`query_logs`（日志查询）、`alert`（告警分析）。
>
> SQL执行有四层安全防护：只读账号 + AST语法解析拦截 + 表级白名单 + 全表扫描拦截。**哪怕AI想搞破坏，它也做不到。**
>
> 而且SQL执行失败后，**系统能自动修复SQL并重试**——不需要人工介入。
>
> 此外，通过**模型路由**策略——简单任务走小模型、复杂推理走强模型——Token成本降低60%。

**幻灯片视觉建议：** 技术栈表格

## 五、Demo展示（2:00）

### 标题：所有角色，所有问题，同一个入口

**幻灯片视觉建议：** 屏幕上方显示问题类型标签栏，随演示切换高亮：`查数据 | 找原因 | 排故障 | 写SQL | 查结构`

**讲述稿：**

> 说了这么多，不如现场演示。而且我要证明的是——**它不只是 Text-to-SQL，所有角色的所有数据问题，都能在这一个对话框里解决。**

### 演示一：【查数据】运营 — "一句话拿数据"（0:25）

> **操作**：输入"近7天的GMV趋势怎么样，有没有异常波动"
>
> **解说**：
>
> 系统自动识别意图 → 调用 `run_sql` 工具 → 执行查询 → 自动分析趋势，**直接告诉你哪天有异常、波动幅度是多少。**
>
> 这个场景，Text-to-SQL 也能做——但区别来了——**我们的系统不仅给你数据，还给你分析结论。**

### 演示二：【找原因】管理者 — "一句话找根因"（0:40）

> **操作**：输入"GMV为什么下降了"
>
> **解说**：
>
> **这个问题，Text-to-SQL 回答不了。** 因为它需要层层下钻。
>
> 看我们的系统怎么做——
>
> （等结果逐层出来）
>
> **它自动做了4层下钻：** 整体对比 → 区域拆分 → 品类拆分 → 订单交叉验证。
>
> 最终给出的不是数据，而是**一个有因果关系的结论**："GMV下降主要由华东地区电子产品销售下滑导致。"
>
> **这不是翻译SQL，这是自动推理。**

### 演示三：【排故障】运维工程师 — "一句话全链路诊断"（0:35）

> **操作**：输入"数据链路报错了，帮我排查一下"
>
> **解说**：
>
> **这个问题更不涉及SQL。** 运维需要的是系统排查，不是数据查询。
>
> 以前排查这种问题：登录Flink控制台 → 登录Kafka管理 → 查日志系统 → 看告警平台。四个系统来回切换，30分钟。
>
> 现在呢？系统自动识别为 TROUBLESHOOT 意图 → 并行调用 `kafka`、`flink`、`query_logs`、`alert` 四个工具 → **同一个对话框，一句话拿到完整的诊断报告。**
>
> **Text-to-SQL 面对这种问题，完全无能为力。**

### 演示四：【SQL诊断】数据工程师 — SQL报错 & 性能问题（0:20）

> **操作1**：输入"这条SQL报错，帮我看看什么问题"
> **操作2**：输入"这条SQL执行很慢，帮我看看什么问题"
>
> **解说**：
>
> 即使是数据工程师的日常——SQL报错诊断、性能优化、表结构查询——也能受益。
>
> 系统自动识别意图 → 查询实际表结构和索引 → 基于真实数据给出分析，而不是LLM凭空编造。
>
> **大家看到了：查数据、找原因、排故障、SQL诊断——5种完全不同的问题类型，5个不同角色的人，同一个入口。**
>
> **这不是 Text-to-SQL。这是一个能解决所有数据相关问题的 AI 助手。**
